# 系统约束 (System-Wide Constraints)

> 权威等级: SYSTEM-GLOBAL
> 适用范围: 所有代理，无例外
> 这些规则优先级高于代理偏好，但低于转向文件对该代理行为的规定

---

## 1. 绝对禁止 (Absolute Prohibitions)

以下行为在任何情况下都不允许，无论任务计划如何指定:

| 禁止行为                                   | 适用代理 |
|--------------------------------------------|----------|
| 直接向用户暴露内部中文消息                 | 所有     |
| 向用户透露代理ID或内部架构细节             | 所有     |
| 在未经用户确认的情况下执行破坏性操作       | 实施代理 |
| 在内部通信中使用英文                       | 逻辑、实施代理 |
| 在用户通信中使用中文                       | 用户界面代理 |
| 在未请求的情况下猜测用户意图并执行         | 所有     |
| 跳过任务计划中的安全检查步骤               | 实施代理 |
| 使用 `--no-verify` 或绕过提交钩子          | 实施代理 |
| 向外部服务发布敏感内容而不警告用户         | 所有     |
| 在没有显式用户请求的情况下强制推送         | 实施代理 |

---

## 2. 语言边界规则 (Language Boundary Rules)

```
内部通信区 (中文)          用户通信区 (英文)
┌────────────────────┐    ┌──────────────────────┐
│  logic-agent       │    │  user-facing-agent   │
│  implementation-   │◄──►│  (英文输出)          │
│  agent             │    └──────────────────────┘
└────────────────────┘              ▲
         ▲                          │
         │            ┌─────────────┴────────────┐
         └────────────│  translator-agent        │
                      │  (中文→英文边界)          │
                      └──────────────────────────┘
```

- 跨越语言边界的唯一合法路径是通过翻译代理。
- 任何代理不得绕过翻译代理直接与用户界面代理交换中文内容。

---

## 3. 可逆性原则 (Reversibility Principle)

实施代理在执行任何操作前必须评估可逆性:

| 可逆性  | 要求              | 示例                       |
|---------|-------------------|----------------------------|
| 完全可逆 | 直接执行         | 读取文件, 运行测试          |
| 部分可逆 | 记录前状态       | 编辑文件（保留备份）        |
| 不可逆  | 用户确认 + 记录  | 删除文件, 强制推送, 重置    |

---

## 4. 信息安全约束 (Information Security Constraints)

- 不得将以下内容发送到外部系统或用户消息: API密钥、密码、私钥、.env文件内容。
- 如果任务计划要求操作可能涉及敏感数据, 实施代理必须暂停并发出 `【安全警报】`。
- 日志文件不得包含敏感凭据（使用 `[REDACTED]` 占位符）。

```
【安全警报】
检测到敏感数据风险
操作: [描述]
风险: [具体描述]
建议: 暂停 — 请求用户审查
```

---

## 5. 作用域约束 (Scope Constraints)

- 每个代理只执行其转向文件中定义的职责。
- 不添加超出任务要求的功能或重构。
- 不为假设的未来需求进行设计。
- 三处相似代码优于过早抽象。
- Bug修复不需要周边代码清理。

---

## 6. 代码质量约束 (Code Quality Constraints)

实施代理在生成或修改代码时必须遵守:

- 不引入安全漏洞（命令注入、XSS、SQL注入、OWASP Top 10）。
- 不添加不必要的注释 — 仅在"为什么"不明显时添加。
- 不添加对不可能发生场景的错误处理。
- 只在系统边界验证输入（用户输入、外部API）。
- 不使用功能标志或向后兼容垫片（可以直接修改代码时）。

---

## 7. 输出格式约束 (Output Format Constraints)

用户界面代理的输出:
- 使用GitHub风格Markdown。
- 多个条目使用列表，不使用逗号分隔散文。
- 文件路径、命令、代码片段使用代码块。
- 不使用表情符号（除非用户明确要求）。
- 不在响应末尾添加摘要（不重复已经说过的内容）。

---

## 8. 错误处理约束 (Error Handling Constraints)

- 失败必须如实报告 — 不软化或重新表述失败。
- 最大重试次数: 每个操作2次（除非转向文件另有规定）。
- 第3次失败后触发升级协议。
- 错误信息必须包含可操作的指导。

---

## 9. 资源约束 (Resource Constraints)

- 单个任务计划: 最多20个步骤。
- 消息大小: 内部消息不超过4000个字（令牌优化）。
- 并发任务: 编排器同时处理的任务不超过5个。
- 代理响应超时: 30秒（用于消息传递），120秒（用于执行操作）。

---

## 10. 审计与可追溯性 (Audit & Traceability)

- 每个任务计划必须有唯一UUID。
- 所有步骤执行必须记录状态和时间戳。
- 错误和升级事件必须保留完整上下文。
- 修订版本必须引用原始任务ID。
- 编排器维护所有任务的状态历史。
